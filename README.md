# where_to_kite

Telegram-бот: по геолокации подсказывает ближайший подходящий кайт-спот и размер кайта, используя прогноз ветра.

## Архитектура

```
src/
  config.js              — вся конфигурация и её валидация в одном месте
  logger.js              — структурированные логи (JSON)
  providers/
    httpClient.js        — fetch с таймаутом и ретраями
    openMeteoProvider.js — источник данных: Open-Meteo (бесплатно, без ключа)
    windyProvider.js     — источник данных: Windy API (платный, для будущего апгрейда)
    windProvider.js      — фабрика + кэш поверх любого источника
  core/                  — чистая бизнес-логика, без Telegram и без сети напрямую
    kiteSize.js           — подбор размера кайта
    spotFinder.js         — поиск и ранжирование спотов
    spots.js               — загрузка и валидация spots.json
    spots.json
    *.test.js              — unit-тесты на чистую логику
  storage/
    userStore.js          — хранилище веса пользователя (файловая БД за интерфейсом)
  bot/
    commands.js           — обработчики команд Telegram
    rateLimit.js           — защита от спама
    format.js               — форматирование ответов
  index.js                 — точка входа, склеивает всё вместе
```

Слой `core/` не знает о Telegram — те же функции можно переиспользовать в MCP-сервере или веб-API, просто вызвав их напрямую.

## Почему так (стабильность / безопасность / расширяемость)

- **Источник данных о ветре взаимозаменяем.** `WIND_PROVIDER=openmeteo|windy` в `.env` — переключение без правки кода. У обоих провайдеров одинаковый контракт (`getWindForecast(lat, lon)`).
- **Кэш поверх провайдера** (`windProvider.js`) — повторные запросы на тот же спот не долбят внешний API. Экономит лимит Open-Meteo (10k запросов/день) и ускоряет ответ.
- **Таймауты и ретраи на HTTP** (`httpClient.js`) — сетевой сбой или зависший запрос не подвешивает бота.
- **`maxSpotsToCheck`** ограничивает, сколько спотов бот реально опрашивает за один вопрос — защита от слишком большой базы спотов и от лишних вызовов API.
- **Rate limit по чату** — защита от спама и от шквала запросов к внешнему API.
- **Валидация `spots.json` при старте** — если база спотов битая (опечатка в координатах, дубль id), бот упадёт сразу с понятной ошибкой, а не тихо сломается посреди работы у пользователя.
- **Персистентное хранилище веса** (`storage/userStore.js`, lowdb) вместо `Map` в памяти — переживает рестарт процесса. Интерфейс абстрагирован: захочешь перейти на Postgres/Redis — меняется только этот файл.
- **Глобальные обработчики ошибок** (`bot.catch`, `unhandledRejection`, `uncaughtException`) — ошибка в одном чате или разовый сетевой сбой не роняют весь бот для всех пользователей.
- **Без `parse_mode: Markdown`** в ответах — спецсимволы в названии спота (`_`, `*`) не ломают отправку сообщения.
- **`.gitignore`** — `.env` и файл с данными пользователей (`db.json`) никогда не попадут в публичный репозиторий.
- **Health-check на HTTP-порту** — стандартный способ для PaaS (Railway и т.п.) понять, что процесс жив.
- **Unit-тесты** на чистую логику (`kiteSize`, `spots`) — `npm test`, без сети, запускаются за секунды.

## Шаг 1. Токен бота
1. Telegram → **@BotFather** → `/newbot`.
2. Сохрани токен вида `123456:ABC-DEF...`.

## Шаг 2. Настроить .env
```bash
cp .env.example .env
```
Впиши `BOT_TOKEN`. Остальное можно не трогать — сработают дефолты (Open-Meteo, радиус 150 км, лимит 5 сек между запросами и т.д.), все они описаны в `.env.example`.

## Шаг 3. Заполнить свои споты
Открой `src/core/spots.json`, замени примеры на свои. Формат:
```json
{
  "id": "unique-id",
  "name": "Название спота",
  "lat": 60.0167,
  "lon": 29.7333,
  "region": "Регион",
  "level": "beginner | intermediate | advanced",
  "type": "flat | wave",
  "goodWindDirections": ["W", "SW", "NW"],
  "minWindMs": 5,
  "maxWindMs": 15
}
```
Если что-то заполнено неверно (например `lat: 200`) — бот при старте укажет точную строку с ошибкой.

## Шаг 4. Запуск и проверка локально
```bash
npm install
npm test        # unit-тесты, без сети
npm start        # сам бот
```
Напиши боту `/start`, потом `/ves 75`, потом пришли геолокацию.

## Шаг 5. Деплой на Railway (бесплатно)
1. https://railway.app → войти через GitHub.
2. Залей проект в свой репозиторий (можно через веб-интерфейс GitHub, без командной строки).
3. Railway: **New Project → Deploy from GitHub repo**.
4. В Variables добавь как минимум `BOT_TOKEN` (остальное — по желанию, см. `.env.example`).
5. Готово, бот работает 24/7.

**Про персистентность на Railway:** без подключённого volume файловая система эфемерна между деплоями — вес пользователей сохранится между рестартами, но обнулится при следующем деплое кода. Для полной надёжности либо подключи Railway volume под `src/storage/`, либо (лучше на будущее) смени `storage/userStore.js` на настоящую БД.

## Та же модель, что у SoarCast.nl (HARMONIE-AROME) — легально и бесплатно
SoarCast.nl (нидерландский кайт/soaring-сервис) использует модель **HARMONIE-AROME** от KNMI (голландская метеослужба) — она официально доступна через Open-Meteo, тем же API, без скрапинга (сам SoarCast прямо запрещает скрапинг без согласования на `soarcast.nl/web/over`). Просто укажи в `.env`:
```
WEATHER_MODEL=knmi_harmonie_arome_netherlands   # NL/Бельгия, 2 км, но всего 2.5 дня вперёд
```
Для французских спотов аналогично доступна модель **AROME** от Météo-France (`meteofrance_arome_france`) — тот же принцип, тот же API. Все варианты и их особенности расписаны в `.env.example`.

Компромисс у обеих моделей один: высокая точность, но короткий горизонт (2–2.5 дня). Дальше Open-Meteo сам подмешивает более грубый глобальный прогноз (ECMWF/ARPEGE) — код это учитывает автоматически, ничего вручную переключать не нужно.

## Апгрейд на Windy API
Если позже захочешь более точные данные: оформи ключ на https://api.windy.com/point-forecast/pricing, впиши `WIND_PROVIDER=windy` и `WINDY_API_KEY` в `.env` — код менять не нужно, интерфейс провайдера тот же.
